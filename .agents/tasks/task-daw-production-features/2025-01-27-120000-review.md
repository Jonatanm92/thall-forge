# WAV export, song structure, and synthesis quality for Thall Forge

Three features land together: offline audio rendering with per-stem WAV export and stereo guitar doubling, song structure enhancements (tempo automation, transitions, DAW marker export, multiple song forms), and a production-quality signal chain overhaul (parallel drum compression, dual-layer bass, guitar articulations, master mix bus, per-instrument volume controls). The implementation mirrors the existing ThallPlayer architecture by rebuilding the full signal chain inside `Tone.Offline` for export, while adding new generation-time metadata (articulations, tempo maps) that thread through from the arranger to both playback and MIDI export.

**Watch for:** The offline renderer duplicates ~350 lines of signal chain construction from `playback.ts` with no shared abstraction, creating a drift risk where future playback changes silently diverge from export output (confirmed). The `lead` stem routes through the guitar open synth but connects to the `leadBus`, meaning lead notes in the offline renderer actually run through `guitarDest` (the cab convolver), which may produce unintended coloration on clean lead lines (confirmed). The `barToTime` function in MIDI export uses base BPM throughout instead of walking the tempo map, so tempo events are placed at incorrect tick positions for songs with breakdown tempo drops (confirmed).

## High-level view

The offline renderer is the centerpiece. It cannot reuse the live `ThallPlayer` instance because `Tone.Offline` creates its own `AudioContext`, so the entire synth graph is reconstructed per render pass. This works but couples the offline module tightly to the live player's implementation details, since any change to one requires a manual mirror in the other.

Song structure gains section transition logic (crash cymbals at section boundaries, drum fills, velocity crescendos) and a tempo map that automatically slows breakdowns by 8-15 BPM. The tempo map feeds into both MIDI export (via header tempo events) and DAW marker generation, though the MIDI export's time calculation has a bug that ignores the tempo map it just wrote.

The synthesis overhaul replaces the former simple bass and single-chain guitar with a dual-layer bass (triangle sub + distorted grit), parallel-compressed drum bus, and a master bus chain (compressor, EQ3 with high-shelf boost, limiter). Guitar articulations (pinch harmonics, hammer-ons, slides) are tagged at generation time in the riff engine and rendered with distinct synth behaviors at playback.

The mix panel and stereo double toggle are UI-only additions that proxy through to the singleton player, but the mix levels are not passed to the offline renderer, so exported WAVs always use the default unity mix regardless of what the user dialed in.

## Offline renderer and signal chain duplication

The `exportWav.ts` module reconstructs the entire `ThallPlayer` signal chain â€” master bus, guitar open/mute, stereo double, pinch harmonic synth, dual-layer bass, drum kit, drum parallel compression â€” inside the `buildChain()` function. The parameters (thresholds, ratios, EQ values, filter frequencies, oscillator configs) are literal copies from `playback.ts`:

```typescript
// playback.ts
const masterComp = new Tone.Compressor({ threshold: -18, ratio: 3, attack: 0.01, release: 0.1 });

// exportWav.ts
const masterComp = new Tone.Compressor({ threshold: -18, ratio: 3, attack: 0.01, release: 0.1 });
```

This isn't just a style issue. The trigger logic is also duplicated: `triggerGuitarNote`, `triggerKickNote`, `triggerSnareNote` in the offline module mirror `triggerGuitar`, `triggerKick`, `triggerSnare` in the player. Any future synthesis change (a new articulation type, a tweaked envelope, an added effect) must be applied in both places or the export won't match playback. With no tests to catch drift, this will diverge.

## Lead routing through the guitar cab convolver

In `buildChain()`, the lead bus is created and connected to the correct destination:

```typescript
const leadBus = new Tone.Gain(1).connect(dest('lead'));
```

But `triggerNote` dispatches lead notes to `chain.guitarOpen`:

```typescript
case 'lead':
  chain.guitarOpen.triggerAttackRelease(midiToNote(note.pitch), note.duration, time, vel * 0.9);
  break;
```

`guitarOpen` is chained through high-pass, distortion, EQ, and the cab IR convolver before reaching `guitarBus`. The `leadBus` gain node exists but receives no signal in the offline renderer. The live player has the same routing (`this.guitarOpen` for lead), so this is consistent behavior â€” but it means lead lines get the full distorted-guitar-cab treatment, which is likely not the intended timbre for melodic leads. The `leadBus` volume control in `MixPanel` has no effect on lead audio in practice (both live and offline).

## MIDI tempo map tick calculation

`songToMidi` writes tempo events using `barToTime` to convert bar numbers to seconds, then `secondsToTicks` for the MIDI header. But `barToTime` uses constant `spBeat = 60 / baseBpm` throughout:

```typescript
function barToTime(bar: number, song: Song): number {
  const baseBpm = song.bpm;
  const spBeat = 60 / baseBpm;
  // ...walks sections using constant spBeat
}
```

For a song at 140 BPM with a breakdown dropping to 128 BPM at bar 12, the tempo event for bar 14 (restore to 140) will be placed as if bars 12-14 were still at 140 BPM, not at 128. The error is small (a few hundred ms) but accumulates with longer breakdowns and produces audible desync when imported into a DAW.

## Mix levels not passed to offline renderer

The `RenderOptions` interface accepts only `stereoDouble`. The `MixPanel` component calls `player.setMixLevels()` on the live player singleton, but the offline renderer's `buildChain()` always creates bus gains at unity (1.0). A user who dials drums down to 60% and exports WAV gets a full-volume drum mix in the file.

## Transition fill collision with existing drum hits

`addTransitionFills` finds the snare or tom track and pushes new hits at `lastBeatStart + s`:

```typescript
for (let s = 0; s < pat.stepsPerBeat; s++) {
  const step = lastBeatStart + s;
  fillTrack.hits.push({ step, ... });
}
```

If the pattern already has snare hits on those steps (common â€” the last beat of a drum pattern often has hits), the result is two hits at the same step with different velocities. The playback engine triggers both, producing a double-strike. The fill should either replace existing hits at those steps or skip steps that are already occupied.

## No test coverage

The project has no test framework. All three features â€” offline rendering, tempo automation, articulation tagging â€” are verified only by successful compilation. The arranger's `applyDynamicBuilds` mutates hit velocities in place; the transition fills append hits to existing tracks. Both are pure-function operations that would benefit from unit tests asserting expected velocity ranges and hit counts.
